Общие

[+] Для улучшения поддержки списка значений добавить в тип opt_option_list_t функцию getValueListFlag ( чтобы четко определять, используется ли для параметра список значений), а также функцию getListItem для получения элементов списка значений в SQL ( аналогично opt_option_list_t.getValue).

[+] Для копирования параметров добавить в тип opt_option_list_t функцию copyOption ( могла был использоваться в модуле Scheduler при копировании параметров в случае изменения модуля, к которому относится пакетное задание).

[+] Возможно, было бы логично при изменении значения флага test_prod_sensitive_flag переносить текущее используемое значение в зависимости от типа текущей БД ( в пром.  БД переносить общее значение в промышленное и наоборот, в тестовых БД переносить общее значение в тестовое и наоборот).

Завершение доработок

[*] После примерно года эксплуатации версии 3.0.0 в очередной версии:

  • удалить из типа opt_option_list_t устаревшие функции а также используемый для обеспечения совместимости параметр moduleOptionName ( функции setDate, setNumber, setString) и атрибут oldModuleName
  • добавить в уникальный ключ по параметрам ( таблица opt_option_new) тип объекта ( поле object_type_id), чтобы дать возможность задания в рамках модуля разных параметров для одноименных объектов разного типа;
  • удалить из пакета pkg_Option устаревшие функции, а также часть реализации процедуры pkg_Option.deleteOption, выполняющую удаление из устаревших таблиц ( было оставлено для совместимости);
  • удалить устаревшие таблицы doc_mask, doc_storage_rule, opt_option, opt_optionv_value и связанные с ними объекты, после чего переименовать opt_option_new в opt_option, opt_option_value_seq в opt_value_seq;
  • удалить поля и ограничения из новых таблиц opt_option_new, opt_value ( и исторических таблиц), связанные с поддержкой устаревших таблиц;
